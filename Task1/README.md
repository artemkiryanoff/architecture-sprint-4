# Задание 1. Планирование: анализ, идентификация проблем и поиск решений. 

## Проанализируйте схему и описание системы
 - отсутствует мониторинг
   не возможно в историческом разрезе увидеть показатели технических и бизнес-метрик, чтобы детально разобраться в текущей ситуации.
 
 - вероятно есть проблемы с высокой утилизацией API и интерфейса
   нет информации поведения пользователей и ботов на стороне API,
   не понятно какое кол-во заказов проходит через API, а какое через GUI,
   нет информации о пользовательском опыте,
   возможны проблемы на стороне БД, например неоптимальные и дорогостоящие запросы, отсутствие реплики и т.д.
 
 - соблюдаются ли микросервисные паттерны
   используются ли транзакционные границы между MES и CRM,
   как API обрабатывает отказоустойчивость, повторы и дедупликацию запросов?


## Разработайте инициативы, которые необходимы для устранения нежелательных ситуаций

 - Внедрение расширенного мониторинга API
   Фиксация успешных и неуспешных запросов к API.
   Логирование времени ответа и ошибок.
   Включение трассировки запросов через OpenTelemetry.

 - Настройка бизнес-метрик для заказов
   Ввод метрики: процент заказов через API vs. через сайт.
   Анализ поведения пользователей в API (количество клиентов, частота заказов и т.д.).

 - Сбор технических метрик API
   Коды ответов HTTP (5хх, 4хх и т.д.).
   Время ответа на запросы.
   Количество запросов в секунду.
   Количество активных клиентов API.

 - Выбор и настройка системы мониторинга (prometheus, ELK, zabbix, grafana и др.)
   Настройка инфраструктуры для хранения и анализа логов.
   Визуализация данных.
   Оповещения, отчеты.

 - Настройка алертов на критические сбои API
   Если процент ошибок превышает порог – уведомление в telrgram, e-mail, создание задачи в Jira.
   Если API медленно отвечает – оповещение SRE/DevOps/разработчиков.

 - Анализ нагрузки и масштабируемости API
   Можно ли выдерживать увеличение нагрузки без падений?
   Нужны ли горизонтальное масштабирование или кэширование?
   Проведение нагрузочного тестирования API.
   Возможно, использование Rate Limiting и Circuit Breaker.

 - Улучшение контрактов API
   Введение версионирования API.
   Описание контрактов с помощью OpenAPI.
   Введение backward-compatible изменений.

 - Добавление кэша и очереди
   Например, для новых заказов добавить кэширование с паттерном Refresh-ahead, либо - Write-Behind.
   RabbitMQ или Kafka.

 - Оптимизация интерфейса MES
   Анализ скорости выполнения запросов.
   Проверка индексов и оптимизация SQL-запросов.
   Создание реплики БД для чтения.
   Кэширование результатов.
 

## Расставьте инициативы в порядке приоритета
 - Внедрение мониторинга
 - Оптимизация интерфейса MES
 - Добавление кэша и очереди

### Какой вы видите целевую архитектуру через полгода

 - настроен мониторинг (коллекция метрик, дашборды, алертинг и тд)
 - оптимизирована работа заказов, интерфейса MES, API.
 - есть понимание использования ресурсов в зависимости от волн пользователей.
 - используются отказоустойчивые паттерны для работы вычислительных ресурсов\сервисов.
