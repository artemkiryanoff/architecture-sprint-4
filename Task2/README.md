# Выбор и настройка мониторинга в системе
## Мотивация
    Система «Александрит» изначально использовала Яндекс Метрику для мониторинга пользовательской активности. 
Однако с добавлением API для оформления заказов стало очевидно, что имеющиеся данные больше не дают полной картины происходящего в системе. 
Это приводит к следующим проблемам:
    -	Отсутствие видимости запросов, приходящих через API.
    -	Невозможность оперативного обнаружения проблем с доступностью сервисов.
    -	Сложность в анализе производительности и пропускной способности системы.
    -	Отсутствие информации о нагрузке на базу данных и другие компоненты.
    
    Внедрение мониторинга позволит:
    -	Выявлять узкие места в производительности API.
    -	Реагировать на сбои до того, как они затронут пользователей (проактивный мониторинг).
    -	Оптимизировать использование ресурсов.
    -	Улучшить пользовательский опыт.
    -   Возможность собирать метрики для бизнеса для принятия решений.

## Выбор подхода к мониторингу
    Для разных компонентов системы будут выбраны разные подходы:
    -	API-сервисы: используем подход RED (Requests, Errors, Duration), так как он позволяет мониторить количество запросов, ошибки и задержки в ответах.
    -	Инфраструктура (БД, серверы): используем USE (Utilization, Saturation, Errors), так как он ориентирован на загрузку ресурсов и помогает выявлять проблемы с их нехваткой.
    -	Общий мониторинг: применяем концепцию «Четырех золотых сигналов» (latency, traffic, errors, saturation) для комплексного контроля над работой системы.
## Метрики и их назначение
    API-сервисы:
    -	Количество запросов (RPS) для интернет-магазина, CRM и MES API.
  		Позволит понимать нагрузку на систему.
                tags: endpoint, user
    -	Количество ошибок HTTP 5хх для API.
    		Поможет оперативно обнаруживать сбои.
		tags: endpoint
    -	Время ответа (latency) для API.
    		Для понимания проблем с производительностью.
		tags: endpoint
    Инфраструктура:
    -	CPU % и Memory Utilization для API и БД.
    		Контроль потребления ресурсов.
		tags: infrastructure, api, DB
    -	Количество соединений с базами данных.
    		Определенит утилизацию ресурсов БД.
		tags: infrastructure, DB
    -	Размер базы данных (shop, MES).
    		Отслеживание роста данных.
		tags: infrastructure, DB
    -	Контроль запросов базы данных (shop, MES).
    		Отслеживание нагруженных\дорогостоящих запросов и последующая оптимизация.
		tags: infrastructure, DB

## План действий
    1.	Развернуть Prometheus, zabbix для сбора метрик и Grafana для визуализации.
    2.	Настроить экспорт метрик из API
    3.	Подключить мониторинг баз данных (PostgreSQL, MySQL).
    4.	Накопить данные (значения метрик) и определить пороговые значения, настроить алерты.
    5.	Настроить дашборды для визуального контроля состояния системы.
    6.	Провести тестирование системы мониторинга, убедиться, что данные собираются корректно.
    7.  Скорректировать пороговые значения и расписать реакции
    8.  Задокументировать в локальной wiki/confluence примененные знания.
