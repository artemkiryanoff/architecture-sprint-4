## Архитектурное решение по кешированию
# Мотивация
    На данный момент MES испытывает проблемы с производительностью, что негативно сказывается на скорости обработки заказов и пользовательском опыте. 
Основные проблемы:
    •	Долгое время загрузки страниц для операторов.
    •	Медленная обработка заказов для клиентов.
    •	Высокая нагрузка на базу данных из-за частых запросов.
    Кеширование поможет:
    •	Уменьшить количество запросов к базе данных.
    •	Ускорить время отклика системы.
    •	Повысить отказоустойчивость.
    Элементы системы, подлежащие кешированию:
    •	Список заказов – загружается часто, но обновляется относительно редко.
    •	Детали заказа – изменяются только при обновлении статуса.
    •	Каталог товаров – редко изменяется, но часто запрашивается.
    •	Статусы заказов – используются в UI, не требуют мгновенного обновления при изменении.
# Предлагаемое решение
    Будет применено серверное кеширование, так как основная проблема связана с нагрузкой на базу данных и временем отклика API.
	
    Выбранный паттерн:
    •	Cache-Aside – система сначала проверяет кеш, и если данные отсутствуют, запрашивает их из базы, а затем сохраняет в кеш.
    •	Причины выбора:
    	Позволяет избежать устаревших данных, так как запись в кеш происходит только при запросе.
    	Гибкость – можно легко менять время жизни кеша.
    	Хорошо подходит для данных с нерегулярными изменениями.
    Отказ от других паттернов:
    •	Write-Through – увеличил бы задержки, так как при каждом изменении необходимо обновлять кеш.
    •	Refresh-Ahead – мог бы нагружать систему ненужными обновлениями.
    Реализация:
    •	Redis – в качестве кеша.
    •	TTL (Time-to-Live) для кеша:
    	Список заказов – 5 минут.
    	Детали заказа – 10 минут или сброс при изменении.
    	Каталог товаров – 1 час.
    	Статусы заказов – 15 минут.
		
	Необходимо использовать паттерн Outbox, чтобы исключить потерю сообщений при отправке в rabbitMQ.

# Инвалидация кеша
    Инвалидация по ключу с использованием RabbitMQ, так как она позволяет оперативно очищать устаревшие данные при изменениях в системе, 
избегая высокой нагрузки на API при массовых обновлениях.
    Стратегия инвалидации	| Преимущества	   		  | Недостатки
    Временная (TTL)	        | Простота реализации		  | Данные могут устаревать до истечения TTL
    По ключу (Key-based)	| Гибкость управления кешем	  | Требует сложной логики зависимостей
    Программная (Manual)	| Обеспечивает актуальные данны   |  Требует явной очистки при обновлении
